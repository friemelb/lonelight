# Phase 2.5: Husky Pre-Push Hooks

## Overview

Implemented Git pre-push hooks using Husky to enforce code quality before commits are pushed to the remote repository. This ensures that broken code, type errors, and failing tests are caught locally before they reach the main branch.

## What Was Implemented

### Husky Installation
- **Version**: 9.1.7 (modern, lightweight)
- **Location**: Root of monorepo
- **Hooks directory**: `.husky/`
- **Auto-setup**: Via `prepare` script in package.json

### Pre-Push Hook Configuration

Automatic checks before every `git push`:

1. **Type Checking** (`npm run type-check`)
   - Validates TypeScript across all workspaces
   - Catches type errors in api, web, and domain packages
   - Fails on: Type errors, unused variables (strict mode)
   - Execution time: ~2-5 seconds

2. **Test Suite** (`npm test`)
   - Runs all 65 tests across workspaces
   - API: 4 tests (health endpoint)
   - Web: 2 tests (App rendering)
   - Domain: 59 tests (validation schemas)
   - Execution time: ~5-10 seconds

**Total Hook Execution Time**: ~7-15 seconds per push

## Why Pre-Push (Not Pre-Commit)

### Decision Rationale

**Pre-Push Advantages:**
- Comprehensive validation without interrupting commit flow
- Developers can make multiple local commits, validated once on push
- 7-15 second delay is acceptable for push operations
- Reduces friction during rapid development iterations
- Allows for quick checkpoint commits during feature development

**Why Not Pre-Commit:**
- Test suite takes 5-10 seconds - too slow for frequent commits
- Would interrupt rapid development flow
- Developers would bypass hooks frequently (`--no-verify`)
- Discourages frequent, small commits (anti-pattern)

**Why Not CI-Only:**
- Issues discovered too late in the workflow
- Broken code reaches remote repository
- More context switching to fix after push
- Slows down team when main branch breaks

## Hook Behavior

### What Runs

**.husky/pre-push** script:

```bash
npm run type-check && npm test
```

**Note**: Husky v9 uses a simplified format without shell script boilerplate. The `&&` operator ensures tests only run if type-check passes (fail-fast behavior).

### Exit Behavior

- **If type-check fails**: Push is blocked, error shown
- **If tests fail**: Push is blocked, error shown
- **If both pass**: Push proceeds normally
- **Exit codes**: Non-zero exit blocks push, zero allows push

### Bypassing Hooks (Emergency Only)

```bash
# Skip pre-push checks (NOT RECOMMENDED)
git push --no-verify
```

**When to bypass:**
- ⚠️ Critical hotfix that must go out immediately
- ⚠️ CI/CD will catch issues anyway
- ⚠️ Working on WIP branch before cleanup
- ⚠️ Hook is broken and needs fixing

**When NOT to bypass:**
- ❌ "I don't want to wait" - Fix your code instead
- ❌ Test failures - These exist for a reason
- ❌ Type errors - These will break production
- ❌ "Just this once" - Slippery slope to broken main branch

### Hook Failure Workflow

1. **Hook detects issue**:
   ```
   > loanlens@0.1.0 test
   > npm run test --workspaces

   FAIL  test/borrower.test.ts > BorrowerRecord Validation

   npm error Lifecycle script `test` failed with error
   ```

2. **Fix the issue**:
   ```bash
   # Make fixes
   npm test  # Verify locally
   ```

3. **Commit fix**:
   ```bash
   git add .
   git commit -m "fix: correct borrower validation test"
   ```

4. **Push again**:
   ```bash
   git push  # Hooks run again, should pass
   ```

## Troubleshooting

### Hook Not Running

**Symptom**: Push completes without running checks

**Solution**:
```bash
# Reinstall hooks
npm run prepare

# Or manually
npx husky install

# Verify hook exists
ls -la .husky/pre-push
```

### Hook Permission Denied

**Symptom**: `Permission denied: .husky/pre-push`

**Solution**:
```bash
chmod +x .husky/pre-push
```

### Tests Pass Locally But Fail in Hook

**Possible Causes**:
- Stale dependencies
- Outdated build artifacts
- Wrong branch
- Environment differences

**Solution**:
```bash
# Clean and reinstall
npm run clean
npm install

# Verify tests
npm test
npm run type-check

# Try push again
git push
```

### Hook Takes Too Long

**Current**: 7-15 seconds (acceptable)

**If it grows beyond 30 seconds**, consider:
- Running only tests in changed packages
- Using `--changedSince` with Git
- Moving slower tests to CI only
- Using jest/vitest watch mode locally

## Monorepo Considerations

### Workspace Testing

- **Hook runs at root level**: Single hook for entire monorepo
- **Tests ALL workspaces**: apps/api, apps/web, packages/domain
- **No workspace-specific hooks needed**: Centralized quality control
- **Single failure blocks push**: Strict quality enforcement
- **Consistent across packages**: Same standards everywhere

### Commands Used

```json
{
  "scripts": {
    "type-check": "npm run type-check --workspaces",
    "test": "npm run test --workspaces"
  }
}
```

Both commands automatically run across all workspaces:
- `apps/api` - API type-check and tests
- `apps/web` - Web type-check and tests
- `packages/domain` - Domain type-check and tests

### Future: Selective Testing

If test suite grows large (>30 seconds), consider:

```bash
# Only test changed packages
npm test --workspace=apps/api

# Or use Git to detect changes
git diff --name-only HEAD~1 | grep "apps/api" && npm test --workspace=apps/api
```

## Integration with CI/CD

### Client-Side vs Server-Side

**Husky (Client-Side)**:
- Runs on developer's machine before push
- First line of defense
- Fast feedback loop
- Can be bypassed with `--no-verify`

**CI/CD (Server-Side)**:
- Runs on GitHub Actions/Jenkins/etc. after push
- Final safety net
- Cannot be bypassed
- Runs on every PR and merge

### Both Layers Required

```
Developer Machine (Husky)
    ↓ type-check + test
    ↓ (catches 90% of issues)
    ↓
Remote Repository
    ↓
CI/CD Pipeline
    ↓ type-check + test + build + lint
    ↓ (catches remaining 10%)
    ↓
Production
```

**Why both?**
- Husky catches most issues early (fast feedback)
- CI ensures no one bypassed hooks
- CI runs additional checks (build, lint, security)
- Defense in depth approach

## Known Issues & Warnings

### 1. React Router Future Flags (Non-Blocking)

**Warnings in web tests:**
```
⚠️ React Router Future Flag Warning: React Router will begin wrapping
state updates in `React.startTransition` in v7...
```

**Impact**: Tests pass, but warnings appear in output
**Fix**: Add future flags to BrowserRouter or upgrade to React Router v7
**Priority**: Low (cosmetic)

### 2. React act() Warnings (Non-Blocking)

**Warnings in web tests:**
```
Warning: An update to Dashboard inside a test was not wrapped in act(...)
```

**Impact**: Tests pass, but warnings appear in output
**Fix**: Wrap state updates in `act()` or use async utilities
**Priority**: Low (cosmetic)

## Lessons Learned

### What Worked Well

1. **Pre-push timing**: 7-15 seconds is acceptable, doesn't disrupt workflow
2. **Clear output**: Emoji and colored text make scanning easy
3. **Fail fast**: Type-check before tests saves time on failures
4. **Auto-install**: `prepare` script ensures all devs have hooks

### What to Watch

1. **Test suite growth**: Monitor execution time, optimize if >30s
2. **Bypass frequency**: If devs bypass often, hook is too strict/slow
3. **CI redundancy**: Ensure CI runs same checks (safety net)

## Conclusion

Husky pre-push hooks provide a reliable, automated safety net for code quality without significantly impacting developer workflow. The 7-15 second execution time is acceptable for push operations and catches the vast majority of issues before they reach the remote repository.

**Key Benefits:**
- 🛡️ Prevents broken code from reaching main branch
- ⚡ Fast feedback (7-15 seconds vs waiting for CI)
- 🤖 Automated (no manual checks needed)
- 👥 Consistent across entire team
- 📚 Well-documented for troubleshooting

**Trade-offs Accepted:**
- Small time cost per push (7-15 seconds)
- Can be bypassed (but CI provides backup)
- Requires discipline to not bypass habitually

The benefits far outweigh the costs, and this setup scales well with the monorepo architecture.

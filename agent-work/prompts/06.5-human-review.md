# Human Review Workflow for LoanLens

## Overview

This document describes the human review workflow implementation for LoanLens, which allows manual review and correction of AI-extracted borrower information.

## Design Decisions

### Review Scope
- **Borrower-level status**: Single review status applies to entire borrower record
- All fields are reviewed together as a complete record

### Correction Strategy
- **Versioned history**: Full audit trail with ability to see all previous values
- Original AI-extracted values are preserved alongside human corrections
- Each correction creates a new entry in the audit log

### Authentication
- **Workflow only**: No user authentication system (for now)
- Review actions are timestamped but not attributed to specific users
- Can be extended later to add user authentication

### Status Transitions
- **Auto-pending**: New borrowers automatically start with `pending_review` status
- Status values: `pending_review` → `approved` | `rejected` | `corrected`

## Database Schema

### Borrowers Table (Extended)
```sql
ALTER TABLE borrowers ADD COLUMN review_status TEXT NOT NULL DEFAULT 'pending_review';
ALTER TABLE borrowers ADD COLUMN reviewed_at TEXT;
ALTER TABLE borrowers ADD COLUMN reviewer_notes TEXT;
CREATE INDEX idx_borrowers_review_status ON borrowers(review_status);
```

### Field Corrections Table (New)
```sql
CREATE TABLE field_corrections (
  id TEXT PRIMARY KEY,
  borrower_id TEXT NOT NULL,
  field_name TEXT NOT NULL,
  original_value TEXT NOT NULL,
  corrected_value TEXT NOT NULL,
  original_confidence REAL NOT NULL,
  source_document_id TEXT NOT NULL,
  source_page INTEGER NOT NULL,
  original_evidence TEXT NOT NULL,
  correction_note TEXT,
  corrected_at TEXT NOT NULL,
  FOREIGN KEY (borrower_id) REFERENCES borrowers(id) ON DELETE CASCADE
);
```

### Borrower Review Audit Table (New)
```sql
CREATE TABLE borrower_review_audit (
  id TEXT PRIMARY KEY,
  borrower_id TEXT NOT NULL,
  action TEXT NOT NULL CHECK(action IN ('approved', 'rejected', 'corrected', 'submitted_for_review')),
  previous_status TEXT NOT NULL,
  new_status TEXT NOT NULL,
  notes TEXT,
  action_at TEXT NOT NULL,
  FOREIGN KEY (borrower_id) REFERENCES borrowers(id) ON DELETE CASCADE
);
```

## API Endpoints

### GET /api/borrowers/review-queue
Get borrowers pending review.

**Query Parameters:**
- `limit` (optional): Number of records to return (default: 50)
- `offset` (optional): Pagination offset (default: 0)

**Response:**
```json
{
  "data": [/* array of BorrowerRecord */],
  "pagination": {
    "total": 10,
    "limit": 50,
    "offset": 0,
    "hasMore": false
  }
}
```

### POST /api/borrowers/:id/review
Approve or reject a borrower.

**Request Body:**
```json
{
  "action": "approve" | "reject",
  "notes": "Optional reviewer notes"
}
```

**Response:**
```json
{
  "success": true,
  "borrower": {/* updated BorrowerRecord */}
}
```

### PATCH /api/borrowers/:id/field
Correct a specific field value.

**Request Body:**
```json
{
  "fieldName": "ssn",
  "correctedValue": "123-45-6789",
  "correctionNote": "Original OCR misread as 123-45-67B9"
}
```

**Response:**
```json
{
  "success": true,
  "borrower": {/* updated BorrowerRecord */}
}
```

**Notes:**
- Automatically changes status to `corrected` if not already
- Creates audit log entry
- Preserves original value in `field_corrections` table

### GET /api/borrowers/:id/audit-history
Get complete audit history for a borrower.

**Response:**
```json
{
  "borrowerId": "uuid",
  "auditHistory": [/* array of ReviewAction */],
  "corrections": [/* array of FieldCorrection */]
}
```

## Repository Methods

### BorrowerRepository (New Methods)

#### updateReviewStatus(id, status, notes?)
Update the review status of a borrower.
```typescript
await borrowerRepository.updateReviewStatus(
  borrowerId,
  ReviewStatus.APPROVED,
  'All fields verified'
);
```

#### findByReviewStatus(status, options?)
Find borrowers by review status with pagination.
```typescript
const borrowers = await borrowerRepository.findByReviewStatus(
  ReviewStatus.PENDING_REVIEW,
  { limit: 50, offset: 0 }
);
```

#### getFieldHistory(borrowerId, fieldName)
Get correction history for a specific field.
```typescript
const history = await borrowerRepository.getFieldHistory(
  borrowerId,
  'ssn'
);
```

#### countByReviewStatus(status)
Count borrowers by review status.
```typescript
const count = await borrowerRepository.countByReviewStatus(
  ReviewStatus.PENDING_REVIEW
);
```

### ReviewRepository (New)

#### createFieldCorrection(borrowerId, fieldName, originalField, correctedValue, note?)

```typescript
await reviewRepository.createFieldCorrection(
  borrowerId,
  'ssn',

  '123-45-6789',
  'Corrected OCR error'
);
```

#### createReviewAction(borrowerId, action, previousStatus, newStatus, notes?)

```typescript

  borrowerId,
  'approved',

  'All data verified'

```

#### getAuditHistory(borrowerId)
Get all review actions for a borrower.

#### getCorrectionsForBorrower(borrowerId)
Get all field corrections for a borrower.

#### getRecentReviewActions(limit?)
Get recent review actions across all borrowers.

## Workflow Examples

```javascript

await fetch(`/api/borrowers/${borrowerId}/review`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },

// Backend creates:
```

## Migration Notes

### Automatic Migration
The migration runs automatically when the API starts:
1. Checks if `review_status` column exists
2. If not, adds new columns to `borrowers` table
3. Creates `field_corrections` table
4. Creates `borrower_review_audit` table
5. All existing borrowers get `pending_review` status by default

### Manual Migration
If needed, run:
```typescript
import { getDatabase } from '@/database';
import { migrateToReviewWorkflow } from '@/database/schema';

const db = getDatabase();
migrateToReviewWorkflow(db);
```

## Architecture Notes

### Why Borrower-Level Status?
- Simplifies workflow for reviewers
- Reduces number of decisions needed
- Clear approval/rejection signal
- Can be extended to field-level later if needed

### Why Versioned History?
- Complete audit trail for compliance
- Ability to revert incorrect corrections
- Visibility into correction patterns (for AI improvement)
- Transparency for all changes

### Why No Authentication?
- Faster initial implementation
- Workflow can be tested without auth complexity
- Easy to add later without changing core logic
- Suitable for single-reviewer scenarios

## Support and Maintenance

### Common Issues

**Issue**: Borrower stuck in pending_review after correction
**Solution**: Corrections automatically set status to `corrected`. If needed, manually approve after corrections.

**Issue**: Correction history not showing
**Solution**: Verify field name matches exactly (case-sensitive). Check `field_corrections` table directly.

**Issue**: Migration not running
**Solution**: Check database logs. Manually call `migrateToReviewWorkflow()` if needed.

### Logging
All review actions are logged with:
- Borrower ID
- Action type
- Previous/new status
- Timestamp
- Optional notes

Check audit log with:
```sql
SELECT * FROM borrower_review_audit WHERE borrower_id = ?;
```

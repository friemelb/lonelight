# Phase 5.5: Extraction Observability and Processing Telemetry

**Date:** 2026-05-08
**Agent:** Claude Code
**Goal:** Add comprehensive observability and telemetry for document extraction and processing pipeline

## Overview

This phase adds production-ready observability to the LoanLens document processing system. It introduces structured logging, processing metrics collection, error tracking, and real-time monitoring through an enhanced dashboard UI. The implementation enables developers and operators to understand system performance, troubleshoot issues, and monitor processing success rates.

**Key Features:**
- Structured JSON logging for all processing operations
- Processing metrics stored in database with timing and success tracking
- Real-time metrics API for dashboard visualization
- Enhanced Dashboard UI with 7 metric cards
- Auto-refresh metrics every 30 seconds
- Success rate and average processing time tracking

## Architecture

### 1. Database Schema

Added two new tables to the existing SQLite schema:

#### **processing_metrics**

Tracks performance and success metrics for each processing operation.

```sql
CREATE TABLE processing_metrics (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  document_id TEXT NOT NULL,
  metric_type TEXT NOT NULL,        -- 'parsing', 'chunking', 'extraction', 'ingestion'
  started_at TEXT NOT NULL,
  completed_at TEXT,
  duration_ms INTEGER,
  success INTEGER NOT NULL,         -- 0 or 1 (boolean)
  error_message TEXT,
  metadata TEXT,                    -- JSON for additional context
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE
);

CREATE INDEX idx_metrics_document_id ON processing_metrics(document_id);
CREATE INDEX idx_metrics_type ON processing_metrics(metric_type);
CREATE INDEX idx_metrics_started_at ON processing_metrics(started_at);
CREATE INDEX idx_metrics_success ON processing_metrics(success);
```

**Purpose:**
- Track duration of parsing, chunking, and extraction operations
- Calculate aggregate metrics (avg processing time, success rate)
- Enable performance analysis and bottleneck identification
- Provide historical data for trend analysis

#### **extraction_attempts**

Tracks retry attempts for document extraction (future-proofing for retry logic).

```sql
CREATE TABLE extraction_attempts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  document_id TEXT NOT NULL,
  attempt_number INTEGER NOT NULL,
  started_at TEXT NOT NULL,
  completed_at TEXT,
  status TEXT NOT NULL,             -- 'in_progress', 'success', 'failed', 'retrying'
  error_type TEXT,
  error_message TEXT,
  chunks_processed INTEGER,
  fields_extracted INTEGER,
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE
);

CREATE INDEX idx_attempts_document_id ON extraction_attempts(document_id);
CREATE INDEX idx_attempts_status ON extraction_attempts(status);
CREATE INDEX idx_attempts_started_at ON extraction_attempts(started_at);
```

**Purpose:**
- Track extraction retry attempts (when implemented)
- Monitor which attempts succeed vs fail
- Provide audit trail for extraction operations

### 3. MetricsRepository

```typescript
export class MetricsRepository {
  constructor(private db: Database.Database) {}

  async createProcessingMetric(metric: ProcessingMetric): Promise<void>
  async createExtractionAttempt(attempt: ExtractionAttempt): Promise<void>
  async getMetricsByDocumentId(documentId: string): Promise<ProcessingMetric[]>
  async getExtractionAttempts(documentId: string): Promise<ExtractionAttempt[]>
  async getMetricsSummary(): Promise<MetricsSummary>
  async getRecentErrors(limit: number = 10): Promise<any[]>
}
```

**Key Methods:**

- **`createProcessingMetric`**: Records timing and success data for parsing operations
- **`getMetricsSummary`**: Aggregates data for dashboard (with complex SQL queries)
  - Calculates success rate: `(successful_ops / total_ops) * 100`
  - Calculates average duration: `AVG(duration_ms) WHERE success = 1`
  - Counts documents by status
  - Counts recent errors (last 24 hours)
  - Returns last 10 metrics for timeline view

### 4. Structured Logging Utility

Centralized structured logging (`apps/api/src/utils/logger.ts`):

```typescript
export class Logger {
  static info(message: string, context?: LogContext): void
  static error(message: string, error: Error, context?: LogContext): void
  static metric(metricType: string, value: number, context?: LogContext): void
  static warn(message: string, context?: LogContext): void
  static debug(message: string, context?: LogContext): void
  static startOperation(operation: string, context?: LogContext): number
  static endOperation(operation: string, startTime: number, context?: LogContext): void
}
```

**Output Format (JSON):**

```json
{
  "level": "info",
  "timestamp": "2026-05-08T20:00:03.276Z",
  "message": "Document parsing completed",
  "operation": "parsing",
  "documentId": "f6b7c534-7f4f-46b2-b9ea-19d4519ce626",
  "filename": "loan-notes.md",

  "duration": 3

```

**Benefits:**
- Easy to parse by log aggregation tools (Splunk, ELK stack, etc.)
- Consistent structure across all log messages
- Includes context (documentId, operation, duration, etc.)
- Supports filtering by level and timestamp

### 5. Instrumented Ingestion Flow

Updated `apps/api/src/routes/ingest.ts` to record metrics and emit structured logs:

**Instrumentation Points:**
1. **Ingestion Start:** Log corpus path and start timestamp
2. **Directory Scan:** Log file count after successful scan
3. **Document Creation:** Log database errors with context
4. **Parsing Start:** Log operation start with documentId and filename
5. **Parsing Success:**
   - Log completion with chunk count and duration
   - Record processing metric in database
6. **Parsing Failure:**
   - Log error with stack trace and context
   - Record failure metric in database
7. **Ingestion Complete:** Log summary (total, success, failed, parsed, duration)

**Example Instrumentation:**

```typescript
const parsingStart = Date.now();

try {
  Logger.info('Starting document parsing', {
    operation: 'parsing',
    documentId: doc.id,
    filename: doc.filename
  });

  const parseResult = await parsingService.parseAndChunkDocument(doc);

  if (parseResult.success) {
    const parsingDuration = Date.now() - parsingStart;

    Logger.info('Document parsing completed', {
      operation: 'parsing',
      documentId: doc.id,
      filename: doc.filename,
      chunkCount: parseResult.chunks.length,
      duration: parsingDuration
    });

    // Record metric
    await metricsRepository.createProcessingMetric({
      documentId: doc.id,
      metricType: 'parsing',
      startedAt: new Date(parsingStart),
      completedAt: new Date(),
      durationMs: parsingDuration,
      success: true,
      metadata: {
        chunkCount: parseResult.chunks.length,
        filename: doc.filename
      }
    });
  }
} catch (error) {
  // Error handling with metric recording
}
```

### 7. Frontend Metrics Store

Zustand store for fetching and caching metrics (`apps/web/src/store/metricsStore.ts`):

```typescript
interface MetricsStore {
  metrics: MetricsSummary | null;
  isLoading: boolean;
  error: string | null;
  fetchMetrics: () => Promise<void>;
}

export const useMetricsStore = create<MetricsStore>((set) => ({
  metrics: null,
  isLoading: false,
  error: null,

  fetchMetrics: async () => {
    set({ isLoading: true, error: null });
    try {
      const response = await fetch('/api/metrics/summary');
      const data = await response.json();
      set({ metrics: data, isLoading: false });
    } catch (error) {
      set({ error: error.message, isLoading: false });
    }
  },
}));
```

Metrics refresh every 30 seconds automatically:

```typescript

  fetchMetrics();
```

## Implementation Details

### Logging Best Practices

**✅ DO:**
- Use structured logging (JSON format)
- Include context (documentId, operation, filename)
- Log at appropriate levels (info, warn, error)
- Include duration for timed operations
- Log both start and completion of operations

**❌ DON'T:**
- Use `console.log` for production logging
- Log sensitive information (passwords, tokens)
- Log in tight loops (performance impact)
- Mix string formatting with structured data

### Metrics Collection Strategy

**When to Record Metrics:**
- ✅ After each document parsing operation
- ✅ When operations fail (with error details)
- ✅ For long-running operations (> 100ms)
- ❌ For simple CRUD operations (too granular)
- ❌ For health check requests (too frequent)

**Metric Metadata:**
- Include filename for debugging
- Include chunk count for parsing metrics
- Include error messages for failures
- Keep metadata under 1KB per metric

### Performance Considerations

**Database Indexes:**
- Index on `document_id` for fast lookups
- Index on `metric_type` for filtering
- Index on `started_at` for time-range queries
- Index on `success` for success rate calculations

**Query Optimization:**
- Use aggregate queries (AVG, COUNT) in SQL
- Limit recent metrics to last 10 (pagination later)
- Cache metrics summary for 10 seconds (future enhancement)

**Frontend Performance:**
- Auto-refresh every 30s (not too frequent)
- Use optional chaining (`metrics?.field ?? '-'`)
- Show loading states for better UX

## Known Limitations

1. **No Metric Aggregation:** Metrics are not pre-aggregated, so large datasets may slow down the summary endpoint (future: add aggregation tables)

2. **No Metric Retention Policy:** Metrics accumulate indefinitely (future: add TTL and archival)

3. **No Real-Time Updates:** Dashboard refreshes every 30s, not real-time (future: WebSocket support)

4. **Limited Error Details:** Error endpoint returns only basic information (future: add stack traces, context)

5. **No Extraction Metrics:** `extraction_attempts` table not yet populated (awaits extraction implementation in Phase 06)
